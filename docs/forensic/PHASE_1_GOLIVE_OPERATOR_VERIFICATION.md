# Phase 1 — Go-Live Operator Verification

Generated: 2026-05-02
Branch: `phase-1-golive-operator-verification`

## Scope

This phase keeps the product stable and focuses only on getting the current Amarktai Network app ready for live verification.

It does **not** redesign the website, revive AmarktAI Assistant, add video/music providers, or change future app architecture. Those are Phase 2+ items after the current app is production-proven.

## Fixed in this phase

### 1. Live Readiness now uses runtime truth

`GET /api/admin/system/live-readiness` now reads from `getDashboardRuntimeTruth()` instead of mixing separate GenX/provider/adult helper paths.

This means System Health/Live Readiness now uses the same source as Settings, AI Engine, Media Studio capability truth, Adult Mode, and Repo Workbench prerequisites.

### 2. Adult Mode readiness is no longer checked through stale helper paths

Live Readiness now reports the merged `adultGate` from runtime truth:

- `ready`
- `needs_provider_test`
- `provider_failed`
- `not_wired`
- `global_flag_disabled`

Adult content remains gated and legal/safety exclusions are unchanged.

### 3. Repo Workbench prompt → PR prerequisites are explicitly checked

Live Readiness now checks the operational chain required for the workflow Graeme wants:

`prompt → patch → checks → commit → push → PR → GitHub merge → VPS redeploy`

The readiness API checks:

- GitHub token exists in the same service-vault path as Settings
- Workspace root is writable
- Temporary workspace create/delete works
- Repo Workbench can patch
- Repo Workbench can run checks
- Repo Workbench can push
- Repo Workbench can create PRs

### 4. Added VPS go-live verification script

New script:

```bash
scripts/verify_golive.sh
```

Run on VPS after merge/deploy:

```bash
cd /var/www/amarktai/repo
sudo bash scripts/verify_golive.sh
```

It verifies:

- package exists
- env/system environment presence
- git branch/commit
- Node/npm versions
- repo workspace root exists and is writable
- npm install
- Prisma generate
- lint
- tests
- `NEXT_PRIVATE_BUILD_WORKER=0 npm run build`
- standalone server exists
- static/public assets copied into standalone
- systemd restart
- local health ping
- public site response
- live readiness endpoint

It writes a server-side report:

```text
docs/forensic/LIVE_VPS_GOLIVE_VERIFICATION.md
```

## Go-live rule

The current app is only GO when:

1. This PR is merged.
2. VPS deploy succeeds.
3. `sudo bash scripts/verify_golive.sh` returns zero failures.
4. Live Readiness returns `overall: PASS` or has only approved `DISABLED`/`OPTIONAL` items for video/music/adult/webdock/queue.
5. Repo Workbench is manually tested with one safe prompt that creates a PR.

## Manual Repo Workbench proof prompt

After deploy, use Repo Workbench with a harmless documentation-only task first:

```text
Create a small docs/forensic/REPO_WORKBENCH_LIVE_TEST.md file that records the current date, current branch, and confirms this was generated by Repo Workbench. Do not touch application code.
```

Expected flow:

1. Select this repo.
2. Create a work branch.
3. Choose Cheap or Balanced agent.
4. Generate patch.
5. Review diff.
6. Apply patch.
7. Run `git diff --stat` and `npm test` if practical.
8. Commit.
9. Push.
10. Create PR.
11. Merge in GitHub.
12. Redeploy VPS.

## Verdict

Phase 1 prepares the current app for real go-live verification. It does not claim final production GO until the VPS script and one real Repo Workbench PR pass live.
